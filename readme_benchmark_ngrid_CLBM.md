# Benchmark README for `ngrid` CLBM

This document consolidates the current **operator-validation** and **numerical benchmark** tasks for the multi-grid / `ngrid` Carleman-linearized lattice Boltzmann model (CLBM) workflows in this repository. This document is assisted by GPT 5.4.

It is intended to complement `readme.md` by focusing specifically on:

- theory-to-code validation of the `ngrid` CLBM operators,
- staged verification of collision, streaming, and coupled CLBM constructions,
- numerical benchmark status for D1Q3 and D2Q9 test problems,
- the current validated scope and remaining limitations.

## Primary theory reference

The implemented CLBM construction follows the theory in:

- `Li_et_al.tex` in this repository
- published version: **Xiangyu Li et al., Phys. Rev. Research 7, 013036**
- APS PDF: <https://journals.aps.org/prresearch/pdf/10.1103/PhysRevResearch.7.013036>

Throughout this README, equation references use the labels in `Li_et_al.tex`, which correspond to the published paper.

---

## 1. Benchmark scope

The current benchmark suite spans two distinct layers.

### A. Operator validation

These tests verify that the implemented CLBM operators match the algebraic and spatial constructions in the paper:

- collision-side Carleman decomposition,
- transfer blocks generated by the Leibniz product rule,
- locality / diagonality of the `n`-point collision operators,
- lifted streaming operators,
- assembled collision+streaming Carleman operators,
- first-level RHS and one-step update consistency.

These tests are the strongest evidence that the **implemented operators** match the intended CLBM theory.

### B. Numerical / workflow benchmarking

These tasks verify that actual runs and driver workflows behave consistently and produce measurable error statistics:

- D1Q3 multi-grid CLBM vs semi-discrete `n`-point LBE comparisons,
- truncation-order comparison workflows,
- direct end-to-end D2Q9 Taylor-Green driver benchmarks using `clbm_tg2d_run.jl::main(...)`.

These tests establish the current **empirical behavior** of the code, but they do not by themselves prove long-time quantitative accuracy for all regimes.

## 2. Theory-to-code validation map

The following equations from Li et al. are explicitly targeted by the current validation scripts.

| Paper equation | Meaning in the paper | What is checked in code | Main script(s) |
|---|---|---|---|
| `Eq. (eq:lbm-carleman)` | Polynomial LBE decomposition into streaming plus `F^(1)`, `F^(2)`, `F^(3)` terms | Direct nonlinear collision RHS equals `F1*f + F2*f^[2] + F3*f^[3]` | `src/CLBM/verify_carleman_construction.jl` |
| `Eq. (eq:defiBij)` | Leibniz-rule transfer blocks `A^i_{i+j-1}` | Lifted transfer blocks act exactly like product-rule lifts on Kronecker powers | `src/CLBM/verify_carleman_construction.jl` |
| `Eq. (eq:F_i_alpha)` | Local `n`-point collision blocks at grid point `x_α` | Same-point terms act locally; mixed-point terms vanish appropriately | `src/CLBM/verify_carleman_construction.jl` |
| `Eq. (eq:diagonality-F)` | Collision locality / diagonality in space | Mixed-point quadratic/cubic inputs give zero response; only local products contribute | `src/CLBM/verify_carleman_construction.jl` |
| `Eq. (eq:eq:lbm-carleman-n)` | Spatially discretized `n`-point nonlinear LBE | Direct `n`-point RHS is matched against CLBM first-level RHS | `src/CLBM/verify_streaming_and_coupling.jl` |
| `Eq. (eq:npt-coll)` | Assembled `n`-point collision Carleman matrix | Sparse assembled collision matrix reproduces direct collision RHS on level 1 | `src/CLBM/verify_carleman_construction.jl` |
| `Eq. (eq:npt-stream)` | Streaming Carleman matrix built from lifted streaming blocks | Lifted streaming block action reproduces product-rule streaming on `φ^[2]`, `φ^[3]` | `src/CLBM/verify_streaming_and_coupling.jl` |
| `Eq. (eq:n_point_Carleman)` | Full `n`-point truncated Carleman matrix `C = C_s + C_c` | Coupled collision+streaming Carleman operator reproduces the direct `n`-point RHS and one-step update | `src/CLBM/verify_streaming_and_coupling.jl` |
| `Eq. (eq:V-phi)` | `n`-point Carleman state layout | The benchmark scripts check consistency of the first block and lifted-state layout in practice | `src/CLBM/verify_streaming_and_coupling.jl`, `src/CLBM/verify_tg2d_main_workflow.jl` |
| `Eq. (eq:clbm)` | Linear ODE for the truncated `n`-point CLBM state | Time-marching workflows and direct `main(...)` smoke benchmarks exercise the assembled linear CLBM evolution | `src/CLBM/verify_streaming_and_coupling.jl`, `src/CLBM/verify_tg2d_main_workflow.jl` |

## 3. Detailed operator validation

## 3.1 Collision-side operator validation  
File: `src/CLBM/verify_carleman_construction.jl`

This script performs a staged D2Q9 verification of the **collision-side** Carleman construction, aligned directly with the equations in the paper.

### Stage 1: Kronecker ordering for lifted states
Purpose:
- verify the ordering convention used for `f^[k]`.

What is checked:
- `f^[1] = f`
- `f^[2] = kron(f, f)`
- `f^[3] = kron(f, kron(f, f))`

Why it matters:
- every subsequent transfer-block and lifted-operator check depends on the same Kronecker ordering convention as the paper around `Eq. (eq:lbm-carleman)`.

### Stage 2: Polynomial collision decomposition  
Paper target:
- `Eq. (eq:lbm-carleman)`

What is checked:
- for multiple positive D2Q9 probe states,
  the direct nonlinear collision RHS equals
  `F1 * f + F2 * f^[2] + F3 * f^[3]`
  to tight numerical tolerance.

Interpretation:
- this is the most direct verification that the implemented `F^(1)`, `F^(2)`, `F^(3)` matrices reproduce the paper's polynomial collision decomposition.

### Stage 3: Transfer blocks from the Leibniz rule  
Paper target:
- `Eq. (eq:defiBij)`

What is checked:
- single-point transfer blocks such as `A_2^2`, `A_3^2`, `A_4^2`, `A_3^3`
  reproduce the expected product-rule action on lifted states.

Interpretation:
- this validates the algebraic lift from polynomial collision terms to linear Carleman blocks.

### Stage 4: `n`-point locality of collision operators  
Paper targets:
- `Eq. (eq:F_i_alpha)`
- `Eq. (eq:diagonality-F)`

What is checked:
- for `ngrid = 2`, same-point quadratic and cubic products produce the expected local output,
- mixed-point products produce zero output.

Interpretation:
- this is a clear code-level verification of the spatial locality / diagonality property stated in the paper.

### Stage 5: Assembled `n`-point collision Carleman matrix  
Paper target:
- `Eq. (eq:npt-coll)`

What is checked:
- the sparse assembled `n`-point collision Carleman matrix reproduces the direct `n`-point collision RHS on the first Carleman level.

Interpretation:
- this confirms that the implemented sparse collision matrix is not just blockwise plausible, but reproduces the assembled collision action expected from the paper.

### Bottom-line conclusion from `verify_carleman_construction.jl`

This script provides the clearest paper-aligned evidence that the repository's collision-side CLBM implementation matches the theory in Li et al. at the level of:

- polynomial decomposition,
- transfer-block construction,
- spatial locality,
- assembled `n`-point collision matrix action.

## 3.2 Streaming and coupled collision+streaming validation  
File: `src/CLBM/verify_streaming_and_coupling.jl`

This script validates the **streaming operator**, the **coupled collision+streaming operator**, and the first-level CLBM evolution.

It is organized in two parts:

- D1Q3 staged checks on a small spatial grid,
- D2Q9 staged checks on tiny periodic grids.

## D1Q3 staged validation

### Stage 1: Interleaved spatial state layout
Paper relation:
- consistent with the `φ(x)` layout in `Eq. (eq:phi)` and the lifted state `Eq. (eq:V-phi)`.

What is checked:
- the interleaved `φ` state is assembled site-by-site exactly as expected.

### Stage 2: Base centered-difference streaming stencil
Paper relation:
- streaming operator `S` in `Eq. (eq:eq:lbm-carleman-n)`,
- streaming discussion in the “Carleman-linearized LBE on spatial discretization grids” subsection.

What is checked:
- the raw interleaved D1Q3 streaming matrix matches the periodic centered-difference stencil exactly.

### Stage 3: Lifted streaming operator satisfies the product rule
Paper target:
- streaming transfer blocks corresponding to the `\mathcal B_i^i` construction behind `Eq. (eq:npt-stream)`.

What is checked:
- the lifted streaming blocks reproduce the product-rule action on `φ^[2]` and `φ^[3]`.

Interpretation:
- this is the streaming-side analog of the collision transfer-block checks.

### Stage 4: Streaming-only first level
Paper targets:
- `Eq. (eq:npt-stream)`
- `Eq. (eq:V-phi)`

What is checked:
- the first CLBM level of the lifted streaming operator reproduces the direct streaming-only RHS.

### Stage 5: Coupled collision + streaming RHS
Paper targets:
- `Eq. (eq:eq:lbm-carleman-n)`
- `Eq. (eq:n_point_Carleman)`

What is checked:
- the assembled CLBM operator `(C_sparse - S_sparse)` reproduces the direct nonlinear spatial LBE RHS on level 1.

Interpretation:
- this is one of the most important operator-level validations in the repository.

### Stage 6: One-step explicit Euler update
Paper targets:
- `Eq. (eq:n_point_Carleman)`
- `Eq. (eq:clbm)`

What is checked:
- one explicit Euler update from the direct `n`-point LBE agrees with one explicit Euler update from the first CLBM level.

### Stage 7: Short-time trajectory entry check
Paper targets:
- `Eq. (eq:V-phi)`
- `Eq. (eq:clbm)`

What is checked:
- direct and CLBM histories agree at initialization and first update,
- both histories remain finite over the short run.

Interpretation:
- this bridges pure operator validation and actual short-time trajectory behavior.

## D2Q9 staged validation

### Stage 1: Raw periodic D2Q9 streaming stencil
What is checked:
- the periodic D2Q9 streaming matrix matches the expected centered-difference x/y stencil.

### Stage 2: Lifted periodic streaming operator
Paper target:
- streaming-side product-rule lifting behind `Eq. (eq:npt-stream)`.

What is checked:
- lifted streaming blocks reproduce the product-rule action on `φ^[2]` and `φ^[3]`.

### Stage 3: Streaming-only first level
What is checked:
- the lifted streaming operator reproduces the direct D2Q9 streaming-only RHS on the first Carleman level.

### Stage 4: Coupled collision + streaming first-level RHS
Paper targets:
- `Eq. (eq:eq:lbm-carleman-n)`
- `Eq. (eq:n_point_Carleman)`

What is checked:
- the assembled D2Q9 CLBM operator reproduces the direct first-level RHS.

### Stage 5: One-step explicit Euler update
Paper targets:
- `Eq. (eq:n_point_Carleman)`
- `Eq. (eq:clbm)`

What is checked:
- one explicit Euler update agrees between direct D2Q9 spatial LBE and CLBM first level.

### Bottom-line conclusion from `verify_streaming_and_coupling.jl`

This script verifies that:

- the raw streaming stencil is implemented correctly,
- the lifted streaming operator obeys the product rule,
- the coupled collision+streaming CLBM operator reproduces the direct `n`-point RHS,
- first-level one-step updates are consistent in both D1Q3 and D2Q9.

This is strong evidence that the **assembled operator construction** is correct, even when longer-time or larger-grid numerical accuracy remains an open question.

## 4. Numerical benchmark summary

Operator validation and numerical benchmarking answer different questions.

- Operator validation asks: “Is the CLBM operator built correctly?”
- Numerical benchmarking asks: “How does the resulting CLBM trajectory compare to a chosen reference evolution?”

The current numerical benchmarks are summarized below.

## 4.1 D1Q3 multi-grid numerical benchmark  
Artifacts:
- `src/CLBM/plot_multigrid_domain_average.jl`
- `src/CLBM/plot_truncation_order_error_comparison.jl`
- `data/plot_multigrid_domain_average_nt100_summary.txt`

### Important reference-model note

For the multi-grid CLBM workflow, the reference is **not** the exact lattice-shift LBM stream.  
It is the **semi-discrete centered finite-difference `n`-point LBE**, because that is the discretization used by the current multi-grid CLBM operator.

This distinction is essential when interpreting the benchmark.

## 4.3 D2Q9 direct top-level workflow benchmark  
Artifacts:
- `src/CLBM/verify_tg2d_main_workflow.jl`
- `data/verify_tg2d_main_workflow_summary.txt`

This benchmark directly exercises the **top-level** D2Q9 Taylor-Green driver:

- `src/CLBM/clbm_tg2d_run.jl`
- specifically, its `main(...)` workflow

rather than rebuilding operator pieces manually.

### What this benchmark verifies

For each tested case, it checks:

- output array shapes are consistent,
- `phiT_lbe` and `phiT_clbm` have matching dimensions,
- the first block of `VT` matches `phiT_clbm`,
- initial CLBM and reference states agree,
- `dist_abs_err`, `dist_rel_err`, `vel_abs_err`, and `vel_rel_err` are internally consistent with recomputed values,
- all outputs are finite.

This benchmark is therefore primarily a **workflow-integrity** and **driver-consistency** benchmark.

- **workflow validated**,
- **operator construction validated at first level**,

## 5. Current validated status

The following statements are supported by the present benchmark suite.

## 5.1 What is validated

### Collision-side theory implementation
Validated against:
- `Eq. (eq:lbm-carleman)`

- `Eq. (eq:diagonality-F)`

Meaning:
- the implemented collision-side Carleman operators reproduce the polynomial and lifted constructions described in Li et al.

### Streaming-side operator implementation
Validated against:
- streaming construction in the spatial-discretization subsection,
- `Eq. (eq:npt-stream)`

Meaning:
- the raw centered-difference streaming matrices and their lifted Carleman versions behave as expected.

### Coupled `n`-point CLBM operator at first level
Validated against:
- `Eq. (eq:eq:lbm-carleman-n)`
- `Eq. (eq:n_point_Carleman)`
- `Eq. (eq:clbm)`

Meaning:
- the assembled CLBM operator reproduces the direct nonlinear `n`-point RHS at the first Carleman level and gives consistent one-step Euler updates.

### D1Q3 numerical benchmark quality
Meaning:
- D1Q3 multigrid CLBM compares well to the semi-discrete centered-difference `n`-point LBE over the benchmarked run.

### D2Q9 top-level workflow integrity
Meaning:
- the D2Q9 Taylor-Green driver `main(...)` now completes direct smoke benchmarks with consistent outputs and finite histories.

---

## 5.2 What is not yet fully validated

The current benchmark suite does **not** yet establish the following:

- long-time quantitative D2Q9 accuracy on larger grids,
- converged D2Q9 behavior versus truncation order,
- production regression thresholds for D2Q9 error, runtime, or memory,
- strong quantitative agreement between CLBM and the selected D2Q9 reference over extended runs,
- complete validation of all boundary-aware D2Q9 regimes beyond smoke benchmarks.

---

## 6. Recommended interpretation for developers

When modifying the CLBM implementation, the benchmark suite should be interpreted in this order:

1. **Collision algebra must stay correct**  
   Check `verify_carleman_construction.jl`.

2. **Streaming and coupled first-level RHS must stay correct**  
   Check `verify_streaming_and_coupling.jl`.

3. **End-to-end driver workflows must remain self-consistent**  
   Check `verify_tg2d_main_workflow.jl`.

4. **Numerical error trends should be monitored separately from operator correctness**  
   In particular, D2Q9 driver success does not yet imply quantitatively accurate long-time dynamics.

## 9. Summary

The repository now has a meaningful layered benchmark story:

- **collision construction validated,**
- **streaming construction validated,**
- **coupled first-level `n`-point CLBM operator validated,**
- **D1Q3 numerical comparison strong,**
