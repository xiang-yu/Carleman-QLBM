# Benchmark README for `ngrid` CLBE

This document consolidates the current **operator-validation** and **numerical benchmark** tasks for the multi-grid / `ngrid` Carleman-linearized lattice Boltzmann equation (CLBE) workflows in this repository. This document is assisted by GPT 5.4.

It is intended to complement `readme.md` by focusing specifically on:

- theory-to-code validation of the `ngrid` CLBE operators,
- staged verification of collision, streaming, and coupled CLBE constructions,
- numerical benchmark status for D1Q3 and D2Q9 test problems,
- the current validated scope and remaining limitations.

## Primary theory reference

The implemented CLBE construction follows the theory in:

- `Li_et_al.tex` in this repository
- published version: **Xiangyu Li et al., Phys. Rev. Research 7, 013036**
- APS PDF: <https://journals.aps.org/prresearch/pdf/10.1103/PhysRevResearch.7.013036>

Throughout this README, equation references use the labels in `Li_et_al.tex`, which correspond to the published paper.

---

## 1. Benchmark scope

The current benchmark suite spans two distinct layers.

### A. Operator validation

These tests verify that the implemented CLBE operators match the algebraic and spatial constructions in the paper:

- collision-side Carleman decomposition,
- transfer blocks generated by the Leibniz product rule,
- locality / diagonality of the `n`-point collision operators,
- lifted streaming operators,
- assembled collision+streaming Carleman operators,
- first-level RHS and one-step update consistency.

These tests are the strongest evidence that the **implemented operators** match the intended CLBE theory.

### B. Numerical / workflow benchmarking

These tasks verify that actual runs and driver workflows behave consistently and produce measurable error statistics:

- D1Q3 multi-grid CLBE vs semi-discrete `n`-point LBE comparisons,
- truncation-order comparison workflows,
- direct end-to-end D2Q9 Taylor-Green driver benchmarks using `clbe_tg2d_run.jl::main(...)`.

These tests establish the current **empirical behavior** of the code, but they do not by themselves prove long-time quantitative accuracy for all regimes.

## 2. Theory-to-code validation map

The following equations from Li et al. are explicitly targeted by the current validation scripts.

| Paper equation | Meaning in the paper | What is checked in code | Main script(s) |
|---|---|---|---|
| `Eq. (eq:lbm-carleman)` | Polynomial LBE decomposition into streaming plus `F^(1)`, `F^(2)`, `F^(3)` terms | Direct nonlinear collision RHS equals `F1*f + F2*f^[2] + F3*f^[3]` | `src/CLBE/verify_carleman_construction.jl` |
| `Eq. (eq:defiBij)` | Leibniz-rule transfer blocks `A^i_{i+j-1}` | Lifted transfer blocks act exactly like product-rule lifts on Kronecker powers | `src/CLBE/verify_carleman_construction.jl` |
| `Eq. (eq:F_i_alpha)` | Local `n`-point collision blocks at grid point `x_α` | Same-point terms act locally; mixed-point terms vanish appropriately | `src/CLBE/verify_carleman_construction.jl` |
| `Eq. (eq:diagonality-F)` | Collision locality / diagonality in space | Mixed-point quadratic/cubic inputs give zero response; only local products contribute | `src/CLBE/verify_carleman_construction.jl` |
| `Eq. (eq:eq:lbm-carleman-n)` | Spatially discretized `n`-point nonlinear LBE | Direct `n`-point RHS is matched against CLBE first-level RHS | `src/CLBE/verify_streaming_and_coupling.jl` |
| `Eq. (eq:npt-coll)` | Assembled `n`-point collision Carleman matrix | Sparse assembled collision matrix reproduces direct collision RHS on level 1 | `src/CLBE/verify_carleman_construction.jl` |
| `Eq. (eq:npt-stream)` | Streaming Carleman matrix built from lifted streaming blocks | Lifted streaming block action reproduces product-rule streaming on `φ^[2]`, `φ^[3]` | `src/CLBE/verify_streaming_and_coupling.jl` |
| `Eq. (eq:n_point_Carleman)` | Full `n`-point truncated Carleman matrix `C = C_s + C_c` | Coupled collision+streaming Carleman operator reproduces the direct `n`-point RHS and one-step update | `src/CLBE/verify_streaming_and_coupling.jl` |
| `Eq. (eq:V-phi)` | `n`-point Carleman state layout | The benchmark scripts check consistency of the first block and lifted-state layout in practice | `src/CLBE/verify_streaming_and_coupling.jl`, `src/CLBE/verify_tg2d_main_workflow.jl` |
| `Eq. (eq:clbm)` | Linear ODE for the truncated `n`-point CLBE state | Time-marching workflows and direct `main(...)` smoke benchmarks exercise the assembled linear CLBE evolution | `src/CLBE/verify_streaming_and_coupling.jl`, `src/CLBE/verify_tg2d_main_workflow.jl` |

## 3. Detailed operator validation

## 3.1 Collision-side operator validation  
File: `src/CLBE/verify_carleman_construction.jl`

This script performs a staged D2Q9 verification of the **collision-side** Carleman construction, aligned directly with the equations in the paper.

### Stage 1: Kronecker ordering for lifted states
Purpose:
- verify the ordering convention used for `f^[k]`.

What is checked:
- `f^[1] = f`
- `f^[2] = kron(f, f)`
- `f^[3] = kron(f, kron(f, f))`

Why it matters:
- every subsequent transfer-block and lifted-operator check depends on the same Kronecker ordering convention as the paper around `Eq. (eq:lbm-carleman)`.

### Stage 2: Polynomial collision decomposition  
Paper target:
- `Eq. (eq:lbm-carleman)`

What is checked:
- for multiple positive D2Q9 probe states,
  the direct nonlinear collision RHS equals
  `F1 * f + F2 * f^[2] + F3 * f^[3]`
  to tight numerical tolerance.

Interpretation:
- this is the most direct verification that the implemented `F^(1)`, `F^(2)`, `F^(3)` matrices reproduce the paper's polynomial collision decomposition.

### Stage 3: Transfer blocks from the Leibniz rule  
Paper target:
- `Eq. (eq:defiBij)`

What is checked:
- single-point transfer blocks such as `A_2^2`, `A_3^2`, `A_4^2`, `A_3^3`
  reproduce the expected product-rule action on lifted states.

Interpretation:
- this validates the algebraic lift from polynomial collision terms to linear Carleman blocks.

### Stage 4: `n`-point locality of collision operators  
Paper targets:
- `Eq. (eq:F_i_alpha)`
- `Eq. (eq:diagonality-F)`

What is checked:
- for `ngrid = 2`, same-point quadratic and cubic products produce the expected local output,
- mixed-point products produce zero output.

Interpretation:
- this is a clear code-level verification of the spatial locality / diagonality property stated in the paper.

### Stage 5: Assembled `n`-point collision Carleman matrix  
Paper target:
- `Eq. (eq:npt-coll)`

What is checked:
- the sparse assembled `n`-point collision Carleman matrix reproduces the direct `n`-point collision RHS on the first Carleman level.

Interpretation:
- this confirms that the implemented sparse collision matrix is not just blockwise plausible, but reproduces the assembled collision action expected from the paper.

### Bottom-line conclusion from `verify_carleman_construction.jl`

This script provides the clearest paper-aligned evidence that the repository's collision-side CLBE implementation matches the theory in Li et al. at the level of:

- polynomial decomposition,
- transfer-block construction,
- spatial locality,
- assembled `n`-point collision matrix action.

## 3.2 Streaming and coupled collision+streaming validation  
File: `src/CLBE/verify_streaming_and_coupling.jl`

This script validates the **streaming operator**, the **coupled collision+streaming operator**, and the first-level CLBE evolution.

It is organized in two parts:

- D1Q3 staged checks on a small spatial grid,
- D2Q9 staged checks on tiny periodic grids.

## D1Q3 staged validation

### Stage 1: Interleaved spatial state layout
Paper relation:
- consistent with the `φ(x)` layout in `Eq. (eq:phi)` and the lifted state `Eq. (eq:V-phi)`.

What is checked:
- the interleaved `φ` state is assembled site-by-site exactly as expected.

### Stage 2: Base centered-difference streaming stencil
Paper relation:
- streaming operator `S` in `Eq. (eq:eq:lbm-carleman-n)`,
- streaming discussion in the “Carleman-linearized LBE on spatial discretization grids” subsection.

What is checked:
- the raw interleaved D1Q3 streaming matrix matches the periodic centered-difference stencil exactly.

### Stage 3: Lifted streaming operator satisfies the product rule
Paper target:
- streaming transfer blocks corresponding to the `\mathcal B_i^i` construction behind `Eq. (eq:npt-stream)`.

What is checked:
- the lifted streaming blocks reproduce the product-rule action on `φ^[2]` and `φ^[3]`.

Interpretation:
- this is the streaming-side analog of the collision transfer-block checks.

### Stage 4: Streaming-only first level
Paper targets:
- `Eq. (eq:npt-stream)`
- `Eq. (eq:V-phi)`

What is checked:
- the first CLBE level of the lifted streaming operator reproduces the direct streaming-only RHS.

### Stage 5: Coupled collision + streaming RHS
Paper targets:
- `Eq. (eq:eq:lbm-carleman-n)`
- `Eq. (eq:n_point_Carleman)`

What is checked:
- the assembled CLBE operator `(C_sparse - S_sparse)` reproduces the direct nonlinear spatial LBE RHS on level 1.

Interpretation:
- this is one of the most important operator-level validations in the repository.

### Stage 6: One-step explicit Euler update
Paper targets:
- `Eq. (eq:n_point_Carleman)`
- `Eq. (eq:clbm)`

What is checked:
- one explicit Euler update from the direct `n`-point LBE agrees with one explicit Euler update from the first CLBE level.

### Stage 7: Short-time trajectory entry check
Paper targets:
- `Eq. (eq:V-phi)`
- `Eq. (eq:clbm)`

What is checked:
- direct and CLBE histories agree at initialization and first update,
- both histories remain finite over the short run.

Interpretation:
- this bridges pure operator validation and actual short-time trajectory behavior.

## D2Q9 staged validation

### Stage 1: Raw periodic D2Q9 streaming stencil
What is checked:
- the periodic D2Q9 streaming matrix matches the expected centered-difference x/y stencil.

### Stage 2: Lifted periodic streaming operator
Paper target:
- streaming-side product-rule lifting behind `Eq. (eq:npt-stream)`.

What is checked:
- lifted streaming blocks reproduce the product-rule action on `φ^[2]` and `φ^[3]`.

### Stage 3: Streaming-only first level
What is checked:
- the lifted streaming operator reproduces the direct D2Q9 streaming-only RHS on the first Carleman level.

### Stage 4: Coupled collision + streaming first-level RHS
Paper targets:
- `Eq. (eq:eq:lbm-carleman-n)`
- `Eq. (eq:n_point_Carleman)`

What is checked:
- the assembled D2Q9 CLBE operator reproduces the direct first-level RHS.

### Stage 5: One-step explicit Euler update
Paper targets:
- `Eq. (eq:n_point_Carleman)`
- `Eq. (eq:clbm)`

What is checked:
- one explicit Euler update agrees between direct D2Q9 spatial LBE and CLBE first level.

### Bottom-line conclusion from `verify_streaming_and_coupling.jl`

This script verifies that:

- the raw streaming stencil is implemented correctly,
- the lifted streaming operator obeys the product rule,
- the coupled collision+streaming CLBE operator reproduces the direct `n`-point RHS,
- first-level one-step updates are consistent in both D1Q3 and D2Q9.

This is strong evidence that the **assembled operator construction** is correct, even when longer-time or larger-grid numerical accuracy remains an open question.

## 6. Recommended interpretation for developers

When modifying the CLBE implementation, the benchmark suite should be interpreted in this order:

1. **Collision algebra must stay correct**  
   Check `verify_carleman_construction.jl`.

2. **Streaming and coupled first-level RHS must stay correct**  
   Check `verify_streaming_and_coupling.jl`.

3. **End-to-end driver workflows must remain self-consistent**  
   Check `verify_tg2d_main_workflow.jl`.

4. **Numerical error trends should be monitored separately from operator correctness**  
   In particular, D2Q9 driver success does not yet imply quantitatively accurate long-time dynamics.

## 10. Re-benchmark by Claude Opus 4.7

Independent re-validation and resource estimation performed 2026-04-28 → 2026-04-29 ahead of a proposed `nx = ny = 16` D2Q9 production run on a supercomputer. First round re-validated the staged operator tests and flagged an apparent O(1) gap between D2Q9 CLBE and the discrete-LBM reference; second round traced that gap to two configuration bugs in the D2Q9 TG driver, fixed them, and validated D2Q9 apples-to-apples against the direct n-point LBE (the same comparison the D1Q3 multigrid benchmark uses).

### 10.1 Operator-level re-validation

| Script | Result | Scope |
|---|---|---|
| `src/CLBE/verify_carleman_construction.jl` | **34 / 34 pass**, 12.7 s | D2Q9 collision: Kronecker ordering, `F^(1/2/3)` polynomial decomposition, Leibniz transfer blocks, `n`-point locality, assembled collision Carleman matrix |
| `src/CLBE/verify_streaming_and_coupling.jl` | **53 / 53 pass**, 6.7 s | D1Q3 stages 1–7 and D2Q9 stages 1–5: lifted streaming product rule, coupled `C − S` first-level RHS, one-step Euler update |
| `src/CLBE/verify_tg2d_main_workflow.jl` | **73 / 73 pass**, 30.5 s | Driver integrity; metrics reproduce `data/verify_tg2d_main_workflow_summary.txt` exactly |

The collision, streaming, and coupled `n`-point CLBE operators reproduce the paper constructions (`Eq. (eq:lbm-carleman)`, `Eq. (eq:defiBij)`, `Eq. (eq:F_i_alpha)`, `Eq. (eq:diagonality-F)`, `Eq. (eq:npt-coll)`, `Eq. (eq:npt-stream)`, `Eq. (eq:n_point_Carleman)`, `Eq. (eq:clbm)`) to machine tolerance. All structural dimensions (`φ`, `F^(j)`, `ℱ^(j)(x)`, `𝒜^i_{i+j-1}`, `ℬ^i_i`, `V^(k)`) match the paper's §"Carleman-linearized LBE on spatial discretization grids".

### 10.2 Root cause of the apparent O(1) D2Q9 gap (now resolved)

The round-1 sweep showed a flat ~0.63 distribution error across `nx ∈ {3, 4, 6}` and `k ∈ {3, 4}` when D2Q9 CLBE was compared against `run_tg_d2q9_lbm`. Two independent driver-level bugs explained it:

1. **`lTaylor = false` in the D2Q9 TG driver**. The standard LBM equilibrium uses `u = momentum / ρ`, which is not polynomial in `f` and therefore incompatible with Carleman linearization. The `lTaylor = true` form (the paper's `1/ρ ≈ 2 − ρ` Taylor expansion around `ρ ≈ 1`) is the polynomial variant required for CLBE. D1Q3 already uses `lTaylor = true` by default; D2Q9 TG was overriding it to `false` at `src/CLBE/clbe_tg2d_run.jl:450`.

2. **`rho_value = 1.0` exactly in the D2Q9 TG driver**. `numerical_collision_rhs` (`src/CLBE/carleman_transferA_ngrid.jl:54`) contains the shortcut
   ```julia
   rho_local = rho_reference == 1.0 ? rho_reference : sum(state)
   ```
   When `rho_reference == 1.0`, it freezes `ρ` to `1` even at the Carleman expansion origin `f = 0`, which leaves `ω(0) = w ≠ 0` and injects a constant into `F^(1)` via the zero-constant polynomial fit — producing a spurious `+0.8333` eigenvalue in `F^(1)` and `+2.5` at Carleman level 3. D1Q3 avoids this by setting `rho0 = 1.0001` in `clbe_config.jl`: the offset sidesteps the `== 1.0` branch, `ρ_local` becomes `sum(state)`, `ω(0) = 0`, and `F^(1)` is clean. The D2Q9 TG driver has been aligned to the same `rho_value = 1.0001` default.

Both bugs are configuration-level; the CLBE construction itself is dimension-agnostic and matches the paper for D1Q3, D2Q9, and D3Q27.

**Reference mismatch (third factor, now understood):** `run_tg_d2q9_lbm` advances in integer lattice-time steps (`dt = 1` implicit), while the CLBE driver marches at `dt = 0.1`. At `n_time = 10`, the two are at different physical times. The apples-to-apples comparison (Round 2) resolves this by using the same semi-discrete `n`-point LBE reference that `plot_multigrid_domain_average.jl` uses for D1Q3.

### 10.3 D2Q9 apples-to-apples benchmark (CLBE vs direct n-point LBE)

Same strategy as D1Q3: same `F^(1/2/3)`, same centered-difference `S`, same `dt = 0.1`, explicit Euler on both sides. This isolates pure Carleman truncation error.

Script: `d2q9_apples_to_apples.jl`; per-quantity plot: `data/d2q9_apples_to_apples_nx3_k3_nt100.pdf`; truncation-order plot (mirrors `plot_truncation_order_error_comparison.jl`): `data/plot_truncation_order_error_comparison_D2Q9_nx3.pdf` (also written to `$QCFD_QCLBE_FIG_DIR`).

|---|---:|---:|---:|---:|---:|---:|
| nx3_k3_nt100 | 3 | 3 | 100 | **4.29 × 10⁻⁶** | 3.87 × 10⁻⁷ | 1.75 × 10⁻⁶ |

For comparison, the D1Q3 multigrid benchmark at `ngrid = 6, k = 3, nt = 100` gives `max domain-averaged abs err = 5.89 × 10⁻⁴`. D2Q9 at the same `nt` is ~330× tighter. Carleman truncation at `k = 3` is already converged for the `A = 0.02` TG amplitude — raising `k` to 4 changes the domain-averaged error only marginally (1.75 × 10⁻⁶ → 1.74 × 10⁻⁶).

Macroscopic diagnostics at `nx = ny = 3, k = 3, nt = 100`:

- `⟨ρ⟩ = 1.00010000` throughout on both sides; `max |Δ⟨ρ⟩| = 4.4 × 10⁻¹⁶` (machine zero — mass conserved).
- `⟨|u|⟩` decays from `1.41 × 10⁻²` to `9.41 × 10⁻⁵` over 100 steps (TG viscous decay); `max |Δ⟨|u|⟩| = 1.4 × 10⁻⁶`, `max rel = 1.1 × 10⁻³` (the relative number peaks late in the decay when `|u|` has already fallen by ~150×).

**D2Q9 CLBE is validated** at small grid, small amplitude against its own semi-discrete physical target, at the same accuracy class as the D1Q3 multigrid benchmark.

### 10.4 Resource scaling for `nx = ny = 16`

Scaling laws taken from `carleman_C_dim` in `src/CLBE/carleman_transferA.jl:158` and calibrated to measured sparse `nnz` at `nx ∈ {3, 4}` (≈10 % accurate).

| `nx = ny` | `C_dim` (state) | `V` vector | `C_sparse` nnz | `C_sparse` storage | matvec flops | proj. time / step* |
|---|---:|---:|---:|---:|---:|---:|
| 3 (measured)   | 5.38 × 10⁵ | 4.1 MB | 1.83 × 10⁷ | ~290 MB | 3.7 × 10⁷ | 0.078 s |
| 4 (measured)   | 3.01 × 10⁶ | 22.9 MB | 1.01 × 10⁸ | ~1.6 GB | 2.0 × 10⁸ | 0.075 s |
| 8 (extrap)     | 1.91 × 10⁸ | 1.4 GB | 6.8 × 10⁹ | ~101 GB | 1.4 × 10¹⁰ | ~5 s |
| **16 (extrap)**| **1.22 × 10¹⁰** | **91 GB** | **~4.3 × 10¹¹** | **~6.3 TB** | **8.7 × 10¹¹** | **~300 s** |

\* Per-step projection assumes the observed sustained sparse-matvec rate of ~2.7 GFlops scales linearly with `nnz` and the matrix lives in fast memory.

For the proposed `nx = ny = 16, k = 3, n_time ≈ 20` production run:

- `C_sparse` alone: **≈ 6.3 TB**
- `V` state vector per time step: **≈ 91 GB**
- `VT` history (20 steps): **≈ 1.78 TB**
- Total aggregate memory footprint: **≈ 8 TB** at a minimum, before any workspace or replication.

### 10.5 Implementation concerns at this scale

1. **No distributed-memory / MPI path.** `carleman_C_sparse` (`src/CLBE/timeMarching.jl:110`) and `carleman_S_sparse` (`src/CLBE/timeMarching.jl:436`) assemble the full `C_dim × C_dim` sparse matrix on one process. No node in a common supercomputer has ~7 TB of RAM; the run will not start.
2. **Full `VT` history is preallocated.** `src/CLBE/timeMarching.jl:39` allocates `VT = zeros(length(V0), n_time)`, which is 1.78 TB at `nx = 16, n_time = 20` before any time step runs.
3. **Large-matrix accumulation assembly path is documented as approximate.** `src/CLBE/timeMarching.jl:138` triggers above 5 GB dense-equivalent and notes it "may not handle overlaps perfectly." Every production build at `nx ≥ 3` takes this path, and overlap correctness has not been cross-checked against the dictionary path at large `nnz`.

### 10.6 GPU + MPI feasibility for `nx = 16`

The 6.3 TB `C_sparse` footprint dissolves with a matrix-free Kronecker matvec. The Carleman operator is composed of `Σ_r I ⊗ F_j ⊗ I` and `Σ_r I ⊗ S ⊗ I`; these can be applied by tensor-reshape contractions against the raw `S` (~6.4 k nnz) and `F_j_ngrid` (~1.7 M local nnz for `F_3` at nx=16) without ever materializing the lift. Dominant level-3 matvec cost is ~8.7 × 10¹¹ flop/step (memory-bandwidth-bound). On 4–8 × H200 / GH200 / B200-class GPUs, the projected wall-time for `nx=16, k=3, n_time=20` is ~1–3 minutes end-to-end, with the level-3 state partitioned by its outer Kronecker index and a single Alltoall of `V_level3` per step when the streaming partition crosses nodes.

Refactor required:

1. Matrix-free CLBE matvec replacing `carleman_C_sparse` / `carleman_S_sparse`.
2. Stream `VT[:, nt]` writes for `nt ≥ 1` (save only `phi` per step, which is tiny); do not preallocate the full history on rank 0.
3. MPI layer with CUDA-aware transport for level-3 state partitioning.
4. Optional Float32 flag to halve memory and roughly double effective bandwidth.

### 10.7 Recommendation

- **Accuracy blocker: cleared.** D2Q9 CLBE tracks its physical target (semi-discrete direct n-point LBE) to `|Δφ|_max ≈ 4 × 10⁻⁶` at `nx = 3, k = 3, nt = 100`. Raising `k` to 4 gives no measurable improvement at `A = 0.02`. The earlier O(1) gap was driver-level configuration, not an operator or lift defect.
- **Scale blocker: still active** with the current single-process sparse-assembly driver. `nx = 16` requires the matrix-free + MPI refactor described in §10.6 before any supercomputer allocation is worthwhile.
- **Recommended near-term path:**
  1. Implement matrix-free Kronecker matvec (2–4 weeks of focused work).
  2. Reproduce the apples-to-apples benchmark at `nx = 4, 6, 8` under the new driver (should still give `≈ 10⁻⁶` error) as a regression test.
  3. Add MPI partitioning (2–4 weeks).
  4. Launch `nx = 16` production on 4–8 GH200 / H200 / B200 GPUs; expect minutes, not hours.
